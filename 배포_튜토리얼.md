# 🚀 Streamlit Community Cloud 배포 튜토리얼

> **대상**: KDT AI Human 6기 수강생 (비전공자 포함) · **목표**: 내가 만든 Streamlit 앱을 `git` 명령 또는 웹 화면만으로 **인터넷 URL**로 배포한다.
> **예시 앱**: ST2 시각화·EDA 대시보드(펭귄) `apps/m3_penguins.py` (외부 모델 다운로드가 없어 가장 가볍게 배포됨).
> ℹ️ `m3`는 파일명에 남은 내부 모듈 번호입니다 — 교안 번호는 ST2입니다(아래 로드맵 참고).
> 이 폴더(`deploy/`) 안의 `requirements.txt`·`packages.txt`가 그대로 업로드용 준비물입니다.

---

## 0. 왜 배포인가 — 3분 개요

지금까지 앱은 `streamlit run`으로 **내 컴퓨터 안에서만** 열렸습니다. 옆 사람에게 보여주려면 화면 공유를 해야 했죠. 배포는 이 한계를 없앱니다.

```
내 컴퓨터(로컬)          GitHub(코드 보관함)         Streamlit Cloud(갤러리)
apps/m3_penguins.py  ─업로드→  저장소의 파일들  ─읽어감→  https://...streamlit.app
```

> 💡 **비유**: 로컬 실행 = 내 방에 걸어둔 액자(나만 봄). 배포 = 갤러리에 옮겨 건 액자(URL만 알면 누구나, 내 컴퓨터가 꺼져 있어도 봄). GitHub는 그 사이의 **택배 보관함**입니다.

⏱️ **소요**: 계정이 있으면 5~10분. 계정 생성부터면 15분.

## 2. GitHub 계정·저장소 만들기

### Step 1 — 계정 생성 (있으면 건너뛰기)
1. [github.com](https://github.com) → 우측 상단 **Sign up**
2. 이메일 → 비밀번호 → **username** 입력
   - username은 프로필 URL(`github.com/내username`)에 **영구히** 남습니다 — 포트폴리오·이력서에 쓰일 수 있으니 신중히.
3. 이메일로 온 인증 코드 입력 → 요금제는 **Free** 선택.

### Step 2 — 저장소(repository) 생성
1. 로그인 후 우측 상단 **+** → **New repository**
2. Repository name: 예) `my-penguin-app`
3. **Public** 선택 (Private는 무료 플랜 배포에 제약 있음)
4. **Add a README file** 체크 → **Create repository**

> 이 저장소는 **ST1에서 한 번만** 만듭니다. ST2~ST5는 같은 저장소에 파일만 추가로 올립니다.

## 3. 저장소에 파일 올리기 — 두 가지 방법

아래 3-A(git CLI, 기본·추천)와 3-B(웹 업로드, 폴백) 중 하나를 선택하세요. **둘 다 결과는 같습니다** — GitHub 저장소에 파일이 올라갑니다.

### 3-A. git CLI로 올리기 (기본 · 추천)

> 터미널(macOS/Linux)이든 PowerShell(Windows)이든 명령은 **동일**합니다. `git`이 설치돼 있어야 합니다(`수업전_환경체크.md` 참고).

**최초 1회 — 로컬과 GitHub 저장소 연결하기**

> §2에서 **Add a README file을 체크했다면(권장 경로)** 원격 저장소에 이미 커밋이 하나 있는 상태입니다 — 아래 **A) git clone**으로 시작하세요. README 체크를 안 했거나 로컬에서 먼저 `git init`을 진행했다면 **B) git init**을 따라가세요.

**A) git clone으로 시작 (권장 — README 체크한 저장소)**
```bash
cd 프로젝트를_둘_폴더        # 예: cd ~/Documents/내프로젝트
git clone https://github.com/내username/저장소이름.git
cd 저장소이름                # 이 폴더 안에 apps/, requirements.txt 등을 넣는다
git add .
git status                  # 무엇이 올라가는지 커밋 전 반드시 확인 — .env가 보이면 STOP
git commit -m "ST1: 첫 Streamlit 앱"
git push
```

**B) git init으로 시작 (README 체크를 안 했거나 이미 로컬에서 init한 경우)**
```bash
cd 내프로젝트폴더           # apps/, requirements.txt가 있는 폴더로 이동
git init
git add .
git status                  # 무엇이 올라가는지 커밋 전 반드시 확인 — .env가 보이면 STOP
git commit -m "ST1: 첫 Streamlit 앱"
git branch -M main
git remote add origin https://github.com/내username/저장소이름.git
git push -u origin main
```
> ⚠️ 원격에 README가 이미 있으면 `push`가 `! [rejected] main -> main (fetch first)`로 거부됩니다. 이때는 두 저장소 히스토리를 합친 뒤 다시 push하세요.
> ```bash
> git pull origin main --allow-unrelated-histories
> git push -u origin main
> ```

**최초 1회 — 인증(비밀번호 대신 필요)**
- GitHub는 2021년부터 비밀번호로 push를 받지 않습니다. 아래 둘 중 하나로 준비하세요.
  - **Personal Access Token(PAT)**: GitHub → Settings → Developer settings → Personal access tokens → 새 토큰 생성(repo 권한) → `git push` 시 비밀번호 칸에 **토큰**을 붙여넣기.
  - **GitHub CLI**: `gh auth login` 1회 실행 → 브라우저로 인증하면 이후 `git push`가 자동으로 인증됩니다.

**이후 매번 — 변경분 올리기(ST2~ST5에서 반복)**
```bash
git status                  # 뭐가 바뀌었는지 먼저 확인하는 습관
git add .
git commit -m "ST2: 펭귄 대시보드 추가"
git push
```

> ⚠️ **보안**: `.env`·`.streamlit/secrets.toml`은 반드시 `.gitignore`에 있어야 합니다. `git add .` 전에 **`git status`로 이 파일들이 목록에 없는지 확인**하는 습관을 들이세요. 이미 커밋해버렸다면 즉시 키를 재발급(rotate)하세요 — 히스토리에서 지워도 이미 노출된 키는 안전하지 않습니다.

### 3-B. 웹 업로드로 올리기 (git이 낯설면 이 방법)

1. 저장소 페이지에서 **Add file → Upload files**
2. 아래 3파일을 **하나씩 개별 선택**해서 올립니다:
   - `apps/m3_penguins.py`
   - `deploy/requirements.txt`
   - `deploy/packages.txt`
3. 하단 **Commit changes** 클릭.

> 🚨 **폴더째 드래그 금지.** 폴더를 통째로 끌어다 놓으면 `.env`(내 API 키)·`__pycache__`까지 올라갑니다. **`.gitignore` 체크박스는 `git` 명령에서만 작동하고, 웹 업로드 화면은 `.gitignore`를 참조하지 않습니다** — "gitignore 했으니 안전"은 착각입니다. 반드시 **필요한 파일만 개별 선택**하세요.

> 📌 Streamlit Cloud는 `requirements.txt`·`packages.txt`를 **저장소 최상단(루트)** 에서 찾습니다. 위 3파일을 저장소 루트에 올리면 됩니다(앱 파일 경로는 다음 단계에서 지정).

## 4. Streamlit Community Cloud로 배포

1. [share.streamlit.io](https://share.streamlit.io) → **Continue with GitHub**(방금 만든 계정으로 로그인)
2. 처음이면 GitHub가 Streamlit에 저장소 접근 **권한**을 요청 → 승인
3. **Create app** → **Deploy a public app from GitHub**
4. 지정:
   - **Repository**: 방금 만든 저장소 (예: `내username/my-penguin-app`)
   - **Branch**: `main`
   - **Main file path**: `m3_penguins.py` (루트에 올렸다면) 또는 `apps/m3_penguins.py`
5. (선택) **Advanced settings** → Python 버전 지정, API 키가 필요한 앱이면 **Secrets**에 TOML로 입력
6. **Deploy!** 클릭 → 빌드 로그가 뜨고 수 분 대기

배포가 끝나면 `https://내username-my-penguin-app.streamlit.app` 같은 URL이 열립니다. 🎉

## 5. 배포 후 확증 (⭐ 로컬 200 ≠ 클라우드)

**"로컬에서 됐으니 배포도 되겠지"는 함정입니다.** 배포 URL을 실제로 열어 아래를 눈으로 확인해야 "완료"입니다.

- [ ] 앱 URL이 열리고 3탭(📊 EDA / 🌲 모델 성능 / 🔮 예측)이 모두 렌더됨
- [ ] 혼동행렬·차트의 **한글 라벨("예측"/"실제")이 □□로 안 깨짐** ← 깨지면 `packages.txt` 누락
- [ ] 슬라이더를 움직이면 화면이 실시간 갱신됨
- [ ] **Manage app**(우하단) → **logs**에 재시작(reboot) 반복이 없음 ← 있으면 메모리 초과
- [ ] 브라우저 개발자도구(F12) **Console**에 빨간 에러 없음

> **[5/5] ST5에서 최종 배포하는 앱**(20강 앱 업그레이드)도 같은 체크리스트로 확증합니다 — cold-load(첫 방문 시 슬립 해제) 확인까지 포함해서요.

## 6. 트러블슈팅 — 증상 → 원인 → 해결

| 증상 | 원인 | 해결 |
|---|---|---|
| 빌드가 매우 느리거나 메모리 초과 | 전체 `requirements.txt`(torch 등)를 올림 | `deploy/requirements.txt`(ST2 최소 6개)로 교체 |
| 앱이 계속 재시작 (reboot loop) | 무료 티어 메모리 초과 (대략 690MB~2.7GB 범위·변경 가능) | 무거운 패키지 제거 + 모델은 `@st.cache_resource`로 1회만 로드 |
| 한글이 □□로 깨짐 | `packages.txt`에 `fonts-nanum` 누락 | `deploy/packages.txt` 업로드 확인 |
| `ModuleNotFoundError` | `requirements.txt`에 패키지 누락 | 빌드 로그가 지목한 패키지를 requirements에 추가 |
| Secrets 관련 `KeyError` | Cloud Secrets에 키 미입력/철자 불일치 | Advanced settings → Secrets에 TOML로 정확히 입력 |
| `git push`가 인증 실패로 거부됨 | 비밀번호로 push 시도 (2021년부터 미지원) | Personal Access Token 또는 `gh auth login`으로 인증 (§3-A) |
| 12시간 뒤 앱이 잠듦(슬립) | 무트래픽 슬립 = **정상** | 방문자가 열면 수십 초 후 자동 재기동 |

## 7. 다른 앱을 배포한다면 — ST3~ST4 확장

- **[3/5] ST3 딥러닝(`m4_image.py`·`m4_sentiment.py`)**: `torch`·`transformers`가 필요합니다. 이땐 requirements 맨 위에 CPU 휠 인덱스를 반드시 넣으세요 — 없으면 GPU(CUDA) 런타임이 딸려와 용량이 폭증해 배포가 깨집니다.
  ```
  --extra-index-url https://download.pytorch.org/whl/cpu
  torch==2.13.0
  torchvision==0.28.0
  transformers==5.6.2
  ```
  > macOS·Windows 모두 **로컬 기본 설치는 CPU 휠**이라 이 문제가 안 보입니다(CUDA는 Cloud/Linux 기본 설치에서만 딸려옴) — "내 컴퓨터에선 됐는데"가 통하지 않는 대표 지점.
- **[4/5] ST4 에이전트(`m5_chatbot.py`·`m5b_agent_loop.py`, `.env`/API 키를 쓰는 앱, 20강 앱)**: 키를 **코드에 넣지 말고** Cloud의 **Secrets** 설정에만 입력하세요. 업로드 시 `.env`는 절대 선택 금지. ST4 앱은 **키가 없어도 데모 모드로 동작**하도록 설계돼 있으니, Secrets를 아직 안 넣었어도 우선 배포해 데모 모드부터 확인할 수 있습니다.
- **바이브코딩 showcase(포트폴리오)**: 같은 절차이지만 requirements는 반대입니다 — 최소화가 아니라 **전체 구성**이 필요합니다. 자세한 절차는 아래 **§8**을 참고하세요.

각 단계는 §3에서 저장소를 새로 만들지 않고, **같은 저장소에 파일을 추가**해 commit/push(3-A) 또는 업로드(3-B)하면 됩니다.

## 8. 포트폴리오(showcase) 배포 — 전체 requirements 사용

지금까지(§1~§7)는 앱 하나씩(m3 펭귄 대시보드 → ST3~ST4 앱)을 **가볍게** 배포하는 절차였습니다. `showcase/`는 다릅니다 — 강사가 이미 완성해 둔 포트폴리오(테마 토큰·다크 토글, 펭귄·이미지·감성 분류 3개 데모 + tool-calling 에이전트, 키 없이도 데모 모드로 동작)이고, **코드를 고치지 않고 그대로** 내 저장소에 올려 배포합니다.

> ⚠️ **정직한 안내**: `showcase/`가 재사용하는 `apps/` 로직은 이 특강 시작 이전에 이미 완성돼 있던 버전입니다. 오늘 ST2~ST3에서 새로 배운 기능(모델 종류 토글·가려보기 히트맵·20문장 벤치마크)은 아직 `showcase/`에 반영돼 있지 않습니다 — 이 배포는 '디자인 퀄리티가 좋은 완성품'을 보여주는 것이 목적이며, 오늘 만든 최신 기능은 로컬에서 `apps/m3_penguins.py`·`apps/m4_image.py`·`apps/m4_sentiment.py`를 직접 실행해 확인하세요.

**로컬 확인** (반드시 프로젝트 루트에서 실행 — `showcase`가 `apps/`를 패키지로 import하므로 cwd가 중요합니다):
```bash
streamlit run showcase/Home.py
# 또는 python3.11 -m streamlit run showcase/Home.py
```

**올릴 파일**:
| 파일 | 위치 | 비고 |
|---|---|---|

## 부록 — 왜 "최소 requirements"인가

Streamlit Cloud는 `requirements.txt`만 보고 **매번 실행 환경을 처음부터 새로** 만듭니다. 여기에 안 쓰는 무거운 패키지(torch 등)가 있으면:
1. 빌드가 느려지고,
2. 무료 티어 메모리(대략 690MB~2.7GB, 변경 가능)를 잡아먹어 앱이 뜨기도 전에 죽습니다.

그래서 "이 앱이 실제로 `import` 하는 것만" 담는 게 핵심입니다. ST2 펭귄 앱은 `streamlit·pandas·numpy·scikit-learn·seaborn·matplotlib` 6개면 충분합니다(torch 없음 → CPU 인덱스도 불필요).

> ⚠️ **예외**: showcase 포트폴리오처럼 여러 모델을 한 앱에 담을 땐 그 앱이 실제로 쓰는 패키지 전체가 필요합니다 — "최소화"가 아니라 "실제로 import하는 것 전부"가 핵심 원칙이고, showcase는 전체가 곧 최소입니다. §8 참고.
